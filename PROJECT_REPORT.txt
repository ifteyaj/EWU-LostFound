1. EXECUTIVE SUMMARY
================================================================================

The EWU Lost & Found system is a comprehensive web-based application designed 
to facilitate the reporting and recovery of lost and found items within the 
East West University campus. This platform provides a centralized, secure, and 
user-friendly solution that connects students who have lost items with those 
who have found them.

The system implements modern web technologies including PHP, MySQL, JavaScript, 
and responsive CSS to deliver a seamless user experience across all devices. 
Key features include user authentication, item reporting, advanced search and 
filtering, profile management, and real-time item status updates.

================================================================================
2. PROJECT OVERVIEW
================================================================================

2.1 Background
--------------
Lost and found items on university campuses often go unclaimed due to lack of 
a centralized reporting system. Students frequently lose valuable items such as 
ID cards, wallets, mobile phones, and academic materials, while those who find 
items have no efficient way to return them to their rightful owners.

2.2 Purpose
-----------
The EWU Lost & Found system aims to:
- Provide a centralized platform for reporting lost and found items
- Enable efficient matching between lost and found items
- Facilitate secure communication between finders and owners
- Reduce item loss and improve recovery rates on campus
- Create a trusted community-based recovery system

2.3 Scope
---------
The system covers:
- User registration and authentication for EWU students
- Lost item reporting with detailed descriptions
- Found item reporting with location information
- Search and filtering functionality
- User profile management
- Item status tracking
- Responsive design for mobile and desktop access

7. DATABASE DESIGN

7.1 Entity-Relationship Overview
--------------------------------
The database consists of three main entities:
- Users: Stores user account information
- Lost Items: Records of items reported as lost
- Found Items: Records of items reported as found

7.2 Database Schema
------------------

TABLE: users
-----------
Field Name          | Type          | Constraints
--------------------|---------------|----------------------------------
id                  | INT           | PRIMARY KEY, AUTO_INCREMENT
student_id          | VARCHAR(50)   | UNIQUE, NOT NULL
email               | VARCHAR(255)  | UNIQUE, NOT NULL
password_hash       | VARCHAR(255)  | NOT NULL
full_name           | VARCHAR(255)  | NOT NULL
phone               | VARCHAR(20)   | 
avatar              | VARCHAR(255)  | DEFAULT NULL
is_admin            | BOOLEAN       | DEFAULT FALSE
is_verified         | BOOLEAN       | DEFAULT FALSE
verification_token  | VARCHAR(255)  | DEFAULT NULL
reset_token         | VARCHAR(255)  | DEFAULT NULL
reset_expires       | DATETIME      | DEFAULT NULL
last_login          | DATETIME      | DEFAULT NULL
created_at          | TIMESTAMP     | DEFAULT CURRENT_TIMESTAMP
updated_at          | TIMESTAMP     | ON UPDATE CURRENT_TIMESTAMP

TABLE: lost_items
-----------------
Field Name          | Type          | Constraints
--------------------|---------------|----------------------------------
id                  | INT           | PRIMARY KEY, AUTO_INCREMENT
user_id             | INT           | FOREIGN KEY (users.id)
item_name           | VARCHAR(255)  | NOT NULL
category            | VARCHAR(100)  | NOT NULL
description         | TEXT          | NOT NULL
last_location       | VARCHAR(255)  | NOT NULL
date_lost           | DATE          | NOT NULL
image               | VARCHAR(255)  | 
student_name        | VARCHAR(255)  | NOT NULL
student_id          | VARCHAR(50)   | NOT NULL
email               | VARCHAR(255)  | NOT NULL
status              | ENUM          | (pending, claimed, resolved)
created_at          | TIMESTAMP     | DEFAULT CURRENT_TIMESTAMP
updated_at          | TIMESTAMP     | ON UPDATE CURRENT_TIMESTAMP

TABLE: found_items
------------------
Field Name          | Type          | Constraints
--------------------|---------------|----------------------------------
id                  | INT           | PRIMARY KEY, AUTO_INCREMENT
user_id             | INT           | FOREIGN KEY (users.id)
item_name           | VARCHAR(255)  | NOT NULL
category            | VARCHAR(100)  | NOT NULL
description         | TEXT          | NOT NULL
found_location      | VARCHAR(255)  | NOT NULL
date_found          | DATE          | NOT NULL
image               | VARCHAR(255)  | 
finder_name         | VARCHAR(255)  | NOT NULL
finder_id           | VARCHAR(50)   | NOT NULL
email               | VARCHAR(255)  | NOT NULL
status              | ENUM          | (pending, claimed, resolved)
created_at          | TIMESTAMP     | DEFAULT CURRENT_TIMESTAMP
updated_at          | TIMESTAMP     | ON UPDATE CURRENT_TIMESTAMP

7.3 Database Indexes
-------------------
Performance optimization indexes:
- idx_lost_category: Index on lost_items.category
- idx_lost_status: Index on lost_items.status
- idx_lost_date: Index on lost_items.date_lost
- idx_found_category: Index on found_items.category
- idx_found_status: Index on found_items.status
- idx_found_date: Index on found_items.date_found
- idx_users_email: Index on users.email
- idx_users_student_id: Index on users.student_id

7.4 Relationships
----------------
- users.id → lost_items.user_id (One-to-Many)
- users.id → found_items.user_id (One-to-Many)
- ON DELETE SET NULL: If user deleted, items remain but user_id is nullified

10.4 Search Implementation
-------------------------
SQL Query:
```sql
SELECT * FROM lost_items 
WHERE (item_name LIKE ? OR description LIKE ?) 
  AND category = ? 
  AND status = ?
ORDER BY created_at DESC
LIMIT ? OFFSET ?
```

Parameters:
- Prepared statements for security
- Wildcard pattern matching
- Case-insensitive search
- Multiple condition support

10.5 Pagination Logic
--------------------
Variables:
- $items_per_page = 8
- $current_page = $_GET['page'] ?? 1
- $offset = ($current_page - 1) * $items_per_page
- $total_pages = ceil($total_items / $items_per_page)

URL Parameters:
- Maintains search query
- Preserves filter settings
- Page number in URL

10.6 Password Security
---------------------
Hashing:

- Verification: password_verify()

Code:

$is_valid = password_verify($password, $password_hash);

13. CHALLENGES AND SOLUTIONS

13.1 Challenge: Responsive Design Consistency
--------------------------------------------
Problem: 
Ensuring consistent navbar behavior and transitions across all pages while
maintaining smooth scroll-based animations.

Solution:
- Centralized JavaScript in script.js for navbar behavior

13.2 Challenge: Search Query Optimization
----------------------------------------
Problem:
Slow search performance with LIKE queries on large datasets.

Solution:
- Added database indexes on frequently searched columns
- Implemented pagination to limit result sets
- Used prepared statements for query optimization

13.3 Challenge: Image Upload and Storage

Large image files consuming server space and slowing page loads.

Solution:

13.4 Challenge: Form Validation Consistency

Ensuring both client-side and server-side validation work together seamlessly.

Solution:
- Dual-layer validation approach
- JavaScript for instant feedback

The EWU Lost & Found system successfully addresses the problem of lost item
recovery on the East West University campus. Through the implementation of
modern web technologies and user-centric design principles, the platform
provides an efficient, secure, and accessible solution for the university
community.

The project demonstrates proficiency in:
- Full-stack web development (PHP, MySQL, JavaScript)
- Database design and optimization
- User interface and user experience design
- Security best practices
- Responsive web design
- Project planning and execution

-----------------------
The EWU Lost & Found system provides tangible benefits:

- Provides centralized item tracking
